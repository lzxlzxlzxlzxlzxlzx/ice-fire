**选型理由：**

| 技术 | 说明 |
|------|------|
| **FastAPI** | 原生支持 async/await + WebSocket，性能接近 Node.js；Pydantic 做数据校验天然适合游戏数据模型；自动生成 OpenAPI 文档 |
| **Uvicorn** | ASGI 服务器，配合 FastAPI async 能力处理 WebSocket 长连接 |
| **SQLAlchemy 2.0** | Python 最成熟的 ORM，异步支持已完善 |
| **Alembic** | 数据库版本迁移，多人协作时保证数据库结构一致 |
| **Pydantic** | 数据校验 + 序列化/反序列化，前后端接口契约 |

### 2.3 数据层（PostgreSQL + Redis）

| 存储 | 存什么 | 为什么 |
|------|--------|--------|
| **PostgreSQL** | 用户账户、游戏历史记录/回放、阵营/单位/地图模板、排行榜、成就 | 持久化、关系查询、事务支持 |
| **Redis** | 活跃对局的实时游戏状态（内存级读写）、WebSocket 会话管理、临时锁（防止操作竞态条件） | 战棋对局中每个操作都需要快速读写游戏状态，PostgreSQL 行锁 + 磁盘 IO 会成为瓶颈 |

**核心策略：双存储模式**

```
游戏进行中 ──→ 状态存 Redis（Set/Get 微秒级）
                              │
                  每回合结束 ──→ 同步到 PG（持久化兜底）
                              │
游戏结束时 ──→ 全部状态写入 PG，Redis 清理
```

### 2.4 通信协议

| 场景 | 协议 | 说明 |
|------|------|------|
| 用户注册/登录 | REST POST | 一次性请求，无状态 JWT 鉴权 |
| 大厅操作（建房间、设置、聊天） | REST + 轮询 | 低频操作，REST 足够 |
| 匹配队列 | WebSocket | 长连接推送匹配结果 |
| 游戏操作（移动、攻击、结束回合） | **WebSocket** | 低延迟、双向推送，服务器控制时序 |
| 战报回放 | REST GET | 读历史记录，文件型数据 |

**WebSocket 消息格式：**

```json
{
  "type": "ACTION_MOVE",
  "payload": {
    "unit_id": "u_1001",
    "path": [{"q": 5, "r": 3}, {"q": 6, "r": 3}]
  },
  "seq": 42
}
```

服务器回复：

```json
{
  "type": "ACTION_RESULT",
  "payload": {
    "success": true,
    "state_diff": { "u_1001": { "q": 6, "r": 3, "move_points": 2 } }
  },
  "seq": 42
}
```

> 使用 `state_diff`（状态差异）而非全量同步，减少网络传输量。

## 三、数据驱动设计

### 3.1 核心理念

**所有游戏内容（单位、建筑、词条、阵营特性）都是数据，不是代码。**

这意味着：
- 新增一个兵种 → 在管理后台填表，点保存 → 游戏中即可使用
- 调整平衡性 → 修改数值 → 即时生效，无需重新部署
- 引擎只解释规则，不硬编码任何具体单位或词条的效果

```
管理后台 ──→ PostgreSQL（配置表）
                  │
游戏引擎加载 ──→  从配置表读取单位/词条定义，在运行时解释执行
```

### 3.2 配置表体系

所有游戏配置数据以"表"的形式存储在 PostgreSQL 中：

| 配置表 | 内容 | 管理端操作 |
|--------|------|-----------|
| unit_types | 单位类型（名称、基础属性、攻击类型、模型资源） | 增删改属性值 |
| traits | 词条定义（ID、名称、效果类型、效果参数） | 增删改效果参数 |
| unit_traits | 单位与词条关联（单位 + 词条 + 叠层数） | 勾选/配置词条 |
| faction_configs | 阵营特性（资源加成、专属单位、初始关系） | 修改数值加成 |
| terrain_types | 地形定义（移动消耗、防御加成、产出） | 修改地形参数 |
| building_types | 建筑定义（消耗、产出、前置条件） | 增删改属性 |
| tech_tree | 科技/升级定义 | 增删改（如未来需要） |
| balance_configs | 全局平衡参数（伤害保底值、反击削弱值等） | 修改数值 |

**关键设计：词条系统（Trait System）**

词条是游戏中所有"特殊能力"的数据化载体。每个词条由以下结构定义：

```python
class TraitDefinition(BaseModel):
    id: str                        # 唯一标识，如 "dragon_glass_weapon"
    name: str                      # 显示名称，如 "龙晶武器"
    description: str               # 效果描述
    effect_type: str               # 效果类型枚举
    effect_params: dict            # 效果参数，由引擎按 type 解析
    stacking: str                  # 叠层规则: "add" / "replace" / "multiply"
    max_stacks: int                # 最大叠层数
```

**effect_type 枚举示例：**

| effect_type | effect_params 示例 | 引擎解析行为 |
|-------------|-------------------|-------------|
| `STAT_MODIFIER` | `{"target": "attack", "value": 10, "scope": "vs_ww"}` | 攻击对异鬼时 +10 |
| `STAT_MODIFIER` | `{"target": "defense", "value": 5, "scope": "always"}` | 防御常驻 +5 |
| `ON_HIT` | `{"chance": 0.3, "effect": "poison", "damage": 5}` | 30% 概率附加中毒 |
| `ON_KILL` | `{"heal": 20}` | 击杀后回血 20 |
| `AURA` | `{"range": 2, "target": "ally", "stat": "attack", "value": 5}` | 光环：周围 2 格友军攻击 +5 |
| `TERRAIN_BONUS` | `{"terrain": "forest", "defense_bonus": 10}` | 在森林中防御 +10 |
| `COUNTER_MODIFIER` | `{"penalty_reduction": 10}` | 远程被近战攻击时反击削弱减少 10 |

这种设计意味着：

- 新增能力 = 选 effect_type + 填 effect_params，**不需要写代码**
- 引擎在战斗/移动/回合结算时遍历单位的所有词条，按 effect_type 分发到对应的处理逻辑
- 极端情况下如果现有 effect_type 无法表达新能力，需要新增 effect_type（需要动代码），但这应该是极少数

### 3.3 管理后台

管理后台是独立于游戏前端的管理界面，负责所有配置数据的 CRUD。

**技术选型：**

- **方案**：React 独立路由 `/admin/*`，与游戏前端同一代码仓库
- **权限**：仅管理员账号可访问（角色校验中间件）
- **部署**：与游戏前端共用同一份构建产物，Nginx 路由分发

**管理后台功能清单：**

```
Admin Panel
├── 单位管理 (Unit Types)
│   ├── 列表/搜索/筛选
│   ├── 新增单位（填写名称、基础属性、攻击类型、模型ID）
│   ├── 编辑单位属性
│   └── 配置词条（从词条库勾选，填参数）
├── 词条管理 (Traits)
│   ├── 词条库列表
│   ├── 新增词条（选 effect_type，填 effect_params）
│   └── 测试词条效果（模拟计算）
├── 阵营管理 (Factions)
│   ├── 基础属性（资源加成、初始关系）
│   └── 专属单位/建筑关联
├── 地形管理 (Terrains)
│   ├── 移动消耗
│   ├── 防御加成/攻击加成
│   └── 资源产出
├── 建筑管理 (Buildings)
│   ├── 建造消耗
│   └── 产出/效果

│   ├── 伤害保底值

├── AI 自对战管理
│   ├── 启动批量对战
│   └── 查看统计数据

```

> 管理后台在游戏引擎完成后即可开始搭建，后续所有平衡性工作不离开浏览器。

## 四、关键架构决策

### 决策 1：服务器权威（Server-Authoritative）

所有关键游戏逻辑（战斗结算、移动合法性、资源变化）**必须在后端执行**。前端只负责：
- 渲染画面
- 提供即时反馈（如高亮移动范围——本地计算后需送服务器验证）
- 发送玩家操作指令

防止作弊，同时保证多人对战时状态一致。

例外：单机模式（AI 对手）时，前后端可以跑在同一进程内，但仍建议保持逻辑在后端，便于未来切换到多人。

### 决策 2：游戏引擎与 Web 框架解耦

`app/core/` 目录下的游戏引擎是**纯 Python 逻辑**，不依赖 FastAPI、HTTP、WebSocket 等任何网络组件。这意味着：
- 引擎可以独立单元测试（pytest 直接调用）
- 引擎可以被不同前端复用（CLI 版本、AI 自对战测试）
- 职责清晰：引擎只管"规则"，框架只管"传输"

```
测试脚本 ──→ engine（纯逻辑）
机器人对练 ──→ engine
FastAPI ──→ engine
```

### 决策 3：热更新部署

前后端分离意味着：
- 前端打包为静态文件（React build），Nginx 托管
- 后端 FastAPI + Uvicorn 运行
- 开发时前端可用 Vite HMR 热更新，后端用 uvicorn --reload 自动重启

```
Nginx
  ├── /api/*       → 反向代理到 FastAPI
  ├── /ws/*        → 反向代理到 FastAPI WebSocket
  └── /*           → 托管前端静态文件
```

## 五、AI 训练与自对战系统

### 5.1 设计目标

- **批量自对战**：AI vs AI 自动运行成百上千局游戏，无需人工参与
- **平衡性测试**：统计数据（胜率、单位使用率、平均回合数）辅助调参
- **AI 策略迭代**：支持不同 AI 策略的对比，未来可接入强化学习
- **与主流程完全解耦**：自对战是独立沙箱，不影响正常游戏服务

### 5.2 架构设计

│  │ Game      │   │ Game      │   │ Game      │  ... N 个实例   │
│  │ Session 1 │   │ Session 2 │   │ Session N │                │
│  │ AI vs AI  │   │ AI vs AI  │   │ AI vs AI  │                │
│  └─────┬─────┘   └─────┬─────┘   └─────┬─────┘                │
│        │               │               │                        │
│        └───────────────┼───────────────┘                        │
│                        │                                        │
│               ┌────────┴────────┐                               │
│               │  Match Recorder │  ← 收集每局完整日志           │
│               └────────┬────────┘                               │
│                        │                                        │
│               ┌────────┴────────┐                               │

### 5.3 核心组件

#### Batch Runner（批量调度器）

独立于 FastAPI 的 Python 进程，职责：

```
batch_runner/
├── runner.py           # 主调度器：管理并发对局、控制节奏
├── game_factory.py     # 从配置表读取阵营/单位/地图，初始化对局
├── ai_factory.py       # AI 策略注册与实例化
└── recorder.py         # 对局数据记录器
```

工作流程：

```
1. 从 PostgreSQL 读取当前配置（单位、词条、地图模板）
2. 初始化 N 个 GameSession（纯内存，不与 Redis/PG 交互）
3. 每个 Session 两个 AI 玩家轮流决策
4. 每步操作写入结构化日志（JSON Lines）
5. 对局结束，结算数据写入 PostgreSQL
6. 启动下一批对局（循环）
```

**关键技术要点：**

- GameSession 直接使用 `app/core/` 下的引擎，不经过 FastAPI
- 每个 Session 运行在独立线程/进程中，互不干扰
- 默认无界面模式（headless），不渲染任何画面
- 典型场景：100 局 × 3 种 AI 策略组合 = 300 局，后台运行

#### AI Strategy Interface

AI 以策略插件的形式存在，可热插拔：

```python
class AIStrategy(ABC):
    """所有 AI 策略的基类"""
    
    @abstractmethod
    def decide(self, game_state: GameState, faction_id: str) -> Action:
        """根据当前游戏状态做出决策"""
        pass
    
    @property
    @abstractmethod
    def name(self) -> str:
        """策略名称，用于统计标识"""
        pass
```

内置策略示例：

| 策略 | 说明 | 复杂度 |
|------|------|--------|
| `RandomAI` | 随机合法行动（基线测试） | ★ |

## 九、城市建筑与单元格改良系统

### 9.1 两套独立体系

城市内建筑与地图上的改良设施是**两套完全独立的系统**，职责不重叠。

```
城市建筑（City Buildings）        单元格改良（Cell Improvements）
───────────────────────           ──────────────────────────
增强"城市"这个抽象概念             增强"地图上这一个格子"的价值
不关心地图位置                     位置就是一切
城市面板上的属性                   绑定单元格坐标
```

| 维度 | 城市建筑 | 单元格改良 |
|------|---------|-----------|
| 存储位置 | `city.buildings: list[BuildingID]` | `cell.improvement: ImprovementID \| null` |
| 占地 | 不占地图格子 | 占一整格 |
| 可被摧毁 | 城市被占领时可能被毁 | 可被敌方部队拆除 |
| 建造者 | 城市自身消耗资源建造 | 工人/工程单位在地图上建造 |
| 效果范围 | 影响整座城市的所有产出 | 仅影响所在格及其相邻格（如要塞） |

### 9.2 城市建筑清单

| 建筑 | 效果 | 不放在地图上的理由 |
|------|------|------------------|
| 军营 | 该城招募单位经验 +1 | 没有空间意义，开关即可 |
| 马厩 | 该城可训练骑兵单位 | 解锁单位类型 |
| 港口 | 该城可建造海军单位 | 解锁单位类型 |
| 市场 | 该城金币产出 +20% | 城市级别产出修正 |
| 铁匠铺 | 该城训练单位攻击 +2 | buff 在招募时注入单位 |
| 城堡/城墙 | 城市防御 +5，耐久 +100 | 城市独占，不单独占格 |
| 塔楼 | 城市视野范围 +2 | 城市级别 |
| 大学/图书馆 | 该城研究速度 +X（如未来有科技树） | 城市级别 |

### 9.3 单元格改良清单

| 改良 | 效果 | 战略意义 |
|------|------|---------|
| 道路 | 该格移动消耗 -1 | 多格串联才有价值，形成补给线 |
| 桥梁 | 跨过河流无视过河惩罚 | 河流上的战略咽喉 |
| 要塞 | 驻守防御 +10 | 隘口、桥头、关键节点 |
| 哨塔 | 视野 +3 | 控制视野区域 |
| 矿井 | 对应资源产出 +X | 绑定富矿格子 |
| 农场 | 人力产出 +2 | 绑定肥沃格子 |
| 营地 | 每回合 HP 恢复 +10 | 前线补给点 |
| 港口 | 沿海格可登船/下船 | 必须建在沿海格上 |

## 十、模块化设计与事件系统

### 10.1 核心原则

**每个系统只负责自己的逻辑，通过 GameState 和 EventBus 与其他系统通信。**

```
系统之间不直接调用
────────────────
❌ 错误：combat.py 里调 unit.modify_hp()
❌ 错误：movement.py 里调 resource.spend_move_point()

✅ 正确：
   combat 计算 → CombatResult 事件 → EventBus
   → TraitSystem 拦截修改 → 统一写入 GameState
```

### 10.2 Turn Pipeline（回合管道）

每回合按固定流水线顺序执行，每个系统只处理自己的阶段：

```
[new turn]
    │
    ▼
ResourceSystem.collect()       ── 计算收入、扣除维护费
    │
    ▼
UnitSystem.reset_actions()     ── 重置移动力、has_attacked
    │
    ▼
CitySystem.process_growth()    ── 人口增长、建筑进度
    │
    ▼
ImprovementSystem.process()    ── 改良设施产出/耐久恢复
    │
    ▼
StatusSystem.tick_effects()    ── buff/debuff 倒计时、过期移除
    │
    ▼
      等待玩家/AI 输入 ────── ActionRouter
           │
           ▼
    ┌──────────────┬───────────────┐
    ▼              ▼               ▼
CombatSystem  MovementSystem  CitySystem
    │              │               │
    └──────────────┴───────────────┘
           │
           ▼
       EventBus ──→ TraitSystem.intercept()
           │
           ▼
       GameState updated（广播 state_diff）
           │
    ──────→ 回到等待输入 / 进入下回合
```

### 10.3 GameState：唯一的共享可变状态

所有系统通过 GameState 读写数据：

```python
class GameState(BaseModel):
    units: dict[UnitID, Unit]
    cells: dict[tuple[int, int], Cell]
    cities: dict[CityID, City]
    factions: dict[FactionID, FactionState]
    turn_number: int
    phase: TurnPhase
    events: list[GameEvent]       # 本回合已发生的事件日志
```

### 10.4 EventBus + 词条拦截

战斗时不是直接扣血，而是通过事件系统流转：

```python
# combat.py — 只计算结果，不直接修改状态（纯函数）
def calculate(attacker, defender, context) -> CombatResult:
    base_damage = max(1, attacker.attack - defender.defense)
    return CombatResult(
        target_id=defender.id,
        base_damage=base_damage,
        is_counter=False,
    )

# event_bus.py — 分发事件，词条系统拦截
event_bus.emit(CombatResult(...))
# → TraitSystem.on_combat() 读取并修改 result
# → 最终写入 GameState
# → 前端收到 state_diff
```

如果之后要加"暴击机制"：
1. 在 `combat.py` 加 crit 计算（只改这一个文件）
2. 或者新增一个 `effect_type: ON_HIT_CRIT` 通过词条数据配置

### 10.5 核心模块写为纯函数

`combat.py`、`movement.py`、`resource.py` 等核心模块写成纯函数——输入 GameState，输出 Event 列表，不持有任何状态：

```python
def resolve_move(
    state: GameState,
    unit_id: UnitID,
    path: list[tuple[int, int]]
) -> list[GameEvent]:
    """纯函数：输入状态和操作，输出事件列表"""
    ...
```

好处：
- 改动任何一个公式，只需要看这一个文件和它的单元测试
- 纯函数可并发执行（为未来自对战加速做准备）
- 测试不需要 mock，直接传 GameState 实例

### 10.6 各系统的改动影响范围

| 想改什么 | 需要改哪里 | 不会影响到 |
|---------|-----------|-----------|
| 战斗公式 | `combat.py` 的 damage 计算 | 寻路、资源、城市、AI |
| 新增词条效果 | `traits/` 下加 effect_type 处理器 | 所有现有词条 |
| 资源产出公式 | `resource.py` 的 collect 逻辑 | 战斗、移动、城市 |
| 移动力消耗规则 | `movement.py` | 战斗、资源 |
| 城市建筑效果 | `city.py` 的 building 处理 | 核心战斗、寻路 |
| 改良设施效果 | `improvement.py` | 城市建筑、单位词条 |
| AI 决策逻辑 | `ai/` 下的策略文件 | 整个引擎 |
