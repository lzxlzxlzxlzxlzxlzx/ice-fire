# 17 AI 训练计划

## 1. 设计目标

```
短期目标：AI 能完成一局完整游戏（不报错、不卡死）
中期目标：AI 能做出合理决策，通过自对局发现平衡性问题
长期目标：AI 具备战术策略，可调整难度，接入强化学习
```

## 3. AI 策略接口

所有 AI 策略实现统一接口，可互相替换、组合测试。

### 3.1 策略接口定义

```python
class AIStrategy(ABC):
    @abstractmethod
    def decide(self, game_state: GameState, faction_id: str) -> Action:
        """根据当前游戏状态做出一个决策"""
        pass

    @property
    @abstractmethod
    def name(self) -> str:
        """策略名称，用于统计标识"""
        pass

    @property
    def difficulty(self) -> str:
        """难度标签: easy / normal / hard"""
        return "normal"
```

### 3.2 内置策略

| 策略 | 难度 | 实现方式 | 用途 |
|------|------|---------|------|
| RandomAI | 极简单 | 从所有合法行动中随机选一个 | 基线测试、冒烟测试 |
| GreedyAI | 简单 | 优先攻击低血量敌人、占领空城 | 基础对战、快速测试 |
| EconomicAI | 中等 | 优先发展经济（建改良、造建筑），后期暴兵 | 测试经济系统平衡性 |
| MilitaryAI | 中等 | 优先爆兵、主动寻战、集中兵力 | 测试战斗系统平衡性 |
| BalancedAI | 中等 | 经济与军事均衡发展，综合评分决策 | 标准 AI 对手 |
| HeuristicAI | 困难 | 多个维度加权评分（兵力/地形/资源/威胁） | AI 挑战、压力测试 |

### 3.3 决策流程

每个 AI 决策周期执行以下步骤：

```
1. 获取当前 GameState
2. 遍历己方所有单位
3. 对每个单位列出合法操作（移动/攻击/待命）
4. 按策略评分每个操作
5. 选择评分最高的操作执行
6. 检查是否还有单位可操作 → 有则回到 2，无则结束回合
```

## 4. 自对战平台

### 4.1 运行模式

| 模式 | 说明 | 用途 |
|------|------|------|
| 快速测试 | 1 局，速度优先，实时看日志 | 开发期验证 AI 不报错 |
| 批量测试 | N 局（如 100 局），后台运行 | 收集统计数据 |
| 对比测试 | 策略 A vs 策略 B，各 N 局 | 评估策略优劣 |
| 平衡测试 | 固定策略，轮换阵营组合 | 发现阵营平衡性问题 |

### 4.2 批量调度流程

```
1. 从 PostgreSQL 读取当前配置（单位/建筑/改良数据）
2. 初始化 N 个 GameSession（纯内存运行，headless）
3. 每个 Session 创建两个 AI 玩家
4. 每步决策调用 AIStrategy.decide()
5. 每步操作写入结构化日志
6. 对局结束，结算数据写入 PostgreSQL
7. 启动下一批对局 → 循环
```

### 4.3 调度参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| 对局总数 | 一次批量运行多少局 | [待定] |
| 并发数 | 同时运行多少个对局 | [待定] |
| 策略组合 | 每局使用什么 AI 策略 | [待定] |
| 阵营组合 | 每局使用什么阵营 | [待定] |
| 地图 | 使用哪张地图 | [待定] |
| 超时 | 单局最大步数（防止死循环） | [待定] |

## 5. 数据收集

### 5.1 对局记录

每局 AI 对战记录以下数据：

```json
{
  "game_id": "ai_20260617_001",
  "timestamp": "2026-06-17T10:00:00Z",
  "config": {
    "map": "标准_80x60",
    "ai_1": {"faction": "NK", "strategy": "BalancedAI"},
    "ai_2": {"faction": "WL", "strategy": "GreedyAI"}
  },
  "result": {
    "winner": "NK",
    "loser": "WL",
    "total_turns": 45,
    "reason": "conquest"
  },
  "turns": [
    {"turn": 1, "events": [...], "state_snapshot": {...}},
    {"turn": 2, "events": [...], "state_snapshot": {...}}
  ],
  "stats": {
    "units_produced": {"NK": 12, "WL": 8},
    "units_lost": {"NK": 5, "WL": 10},
    "cities_captured": {"NK": 2, "WL": 0},
    "damage_dealt": {"NK": 450, "WL": 320}
  }
}
```

### 5.2 统计指标

| 指标 | 计算方式 | 用途 |
|------|---------|------|
| 胜率 | 某阵营/某策略获胜局数 / 总局数 | 发现不平衡 |
| 平均回合数 | 所有对局的回合数均值 | 评估游戏节奏 |
| 单位出场率 | 某单位被生产的次数 / 总局数 | 发现冷门/过强单位 |
| 单位 K/D | 某单位击杀数 / 阵亡数 | 评估单位强度 |
| 平均交战次数 | 每局发生的战斗次数 | 评估战斗频率 |
| 经济曲线 | 每回合资源量的变化 | 评估经济节奏 |

### 5.3 异常警报

当以下情况发生时，系统自动标记：

| 条件 | 提示 | 处理建议 |
|------|------|---------|
| 某阵营胜率 > 65% | 阵营过强 | 削弱该阵营或增强其对手 |
| 某阵营胜率 < 35% | 阵营过弱 | 增强该阵营 |
| 某单位出场率 < 5% | 单位过弱或性价比低 | 检查属性 |
| 某单位出场率 > 40% | 单位过强或万金油 | 考虑削弱 |
| 平均回合数 > [待定] | 游戏节奏过慢 | 调整移动力/地图大小 |
| 平均回合数 < [待定] | 游戏节奏过快 | 增强城防或恢复能力 |

## 7. 实施路线

### 第一阶段：AI 能动起来

```
目标：AI 能完成一局完整游戏
内容：
  ├── RandomAI 实现（随机选合法操作）
  ├── GreedyAI 实现（简单贪心策略）
  ├── 自对战快速测试模式（1 局）
  └── 基础日志（记录每步操作 + 最终结果）
```

### 第二阶段：批量自对战

```
目标：收集统计数据，发现明显不平衡
内容：
  ├── 批量调度器（支持 N 局后台运行）
  ├── 数据记录器（结构化写入 PostgreSQL）
  ├── 统计报表（胜率/出场率/平均回合数）
  └── 异常警报逻辑
```

### 第三阶段：中等 AI

```
目标：AI 具备基本策略，可作为玩家对手
内容：
  ├── EconomicAI（发展经济→暴兵）
  ├── MilitaryAI（早期爆兵→压制）
  ├── BalancedAI（均衡策略，标准 AI）
  └── 难度分级（Easy/Normal/Hard 对应不同策略）
```

### 第四阶段：高级 AI + 管理面板集成

```
目标：AI 决策质量提升，管理后台可控制自对战
内容：
  ├── HeuristicAI（多维度加权评分）
  ├── 管理后台集成（Web 界面启动自对战、查看报告）
  ├── 配置版本管理（每次调整后自动运行回归测试）
  └── 平衡性仪表盘（可视化统计图表）
```

### 第五阶段：强化学习（远期）

```
目标：AI 通过自我博弈持续进步
内容：
  ├── 状态编码器（GameState → 特征向量）
  ├── 奖励函数设计
  ├── 训练循环（分布式自对战 + 模型更新）
  └── RL 模型与规则 AI 的对比评估
```

## 8. 设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| AI 策略插件化 | 统一接口，可热插拔 | 方便对比不同策略，后期接入 RL 不改框架 |
| 自对战纯内存运行 | 不经过 Redis/PG | 速度快，适合批量运行，与正式服务隔离 |
| 先 Random/Greedy 后高级 | 逐步递进 | 先保证"能跑"，再追求"跑得好" |
| 自动异常警报 | 基于统计阈值 | 减少人工检查成本，批量测试后自动出报告 |
