# 06 回合概念

## 1. 回合结构总览

每回合按以下四个阶段顺序执行：

```
┌──────────────────────────────────────────────────────────┐
│  Phase 1: 回合开始（自动结算，全服同步）                  │
│  ─────────────────────────────────────────────              │
│  资源产出 → 维护费 → 城市恢复 → 城墙修复 → 单位重置 → buff计时 │
├──────────────────────────────────────────────────────────┤
│  Phase 2: 人类玩家行动（同时进行）                         │
│  ─────────────────────────────────────────────              │
│  所有人类玩家在同一时间段内各自操作                        │
│  操作实时提交、实时生效                                    │
│  每个玩家独立结束自己的回合                                │
│  全部人类玩家结束后 → Phase 3                              │
├──────────────────────────────────────────────────────────┤
│  Phase 3: AI 行动（轮流进行）                              │
│  ─────────────────────────────────────────────              │
│  AI-1 完成所有操作 → AI-2 完成所有操作 → ...               │
│  全部 AI 结束后 → Phase 4                                  │
├──────────────────────────────────────────────────────────┤
│  Phase 4: 回合结束（自动结算）                             │
│  ─────────────────────────────────────────────              │
│  检查胜利条件 → 回合数 +1 → 进入下一回合                   │
└──────────────────────────────────────────────────────────┘
```

### 1.1 关键原则

| 原则 | 说明 |
|------|------|
| 人类同时行动 | 所有人类玩家在同一时间段内各自操作，互不等待 |
| AI 轮流行棋 | AI 在人类全部结束后依次行动，每个 AI 完成所有操作后才轮到下一个 |
| 操作即时生效 | 玩家的操作一旦提交到服务器，立即写入 GameState，其他玩家下次同步时可见 |
| 冲突先到先得 | 两个玩家同时操作同一目标时，服务器按收到顺序处理，后续操作失败 |

---

## 2. Phase 1：回合开始

此阶段为自动结算，所有玩家同时看到结果。按以下顺序执行：

```
1. 城市产出结算
   ├── 每个城市计算：基础产出 + 建筑加成 + 改良产出
   └── 产出加入所属阵营的资源池

2. 单位维护费扣除
   ├── 每个单位消耗金币维护费
   └── 金币不足时 → 单位士气下降或解散 [待定]

3. 城市 HP 恢复
   ├── 每个城市自动恢复 HP
   └── 恢复量 [待定]/回合

4. 城墙修复检查
   ├── 城墙是否连续 3 回合未受损？
   ├── 城市是否有足够石材？
   └── 是 → 自动修复 [待定] 点耐久，扣除石材

5. 单位状态重置
   ├── move_points = max_move_points（满移动力）
   ├── has_attacked = false
   └── 状态效果计时 -1（到期者移除）

6. 城市状态重置
   └── has_attacked = false（城墙射击重置）

7. buff/debuff 计时
   └── 每个 StatusEffect.remaining_turns -1，归零时移除
```

## 3. Phase 2：人类玩家行动

### 3.1 同时行动机制

所有人类玩家在**同一个时间段内**各自操作。

```
时间线：
├── 回合开始（所有人看到 Phase 1 结果）
├── 玩家 A 开始操作（玩家 B 也在同时操作）
│   ├── A 移动单位 → 服务器即时处理，A 看到结果
│   ├── B 攻击城市 → 服务器即时处理，B 看到结果
│   ├── A 建造改良 → 服务器即时处理
│   ├── A 结束回合（A 进入观战/等待状态）
│   └── B 继续操作...
├── 玩家 B 结束回合
└── 所有人类玩家结束 → 进入 Phase 3
```

**对冲突的处理：**

```
场景：玩家 A 和玩家 B 试图攻击同一个单位
  → 服务器先收到 A 的请求：A 攻击，单位受伤或死亡
  → 服务器后收到 B 的请求：目标已死亡（或已不在原格），操作无效，返回错误提示
```

### 3.2 玩家可执行的操作

| 操作 | 说明 | 限制 |
|------|------|------|
| 移动 | 选中单位，点击目标格 | 消耗移动力，不可穿过敌方单位 |
| 攻击 | 选中单位，攻击范围内敌方目标 | 每单位每回合一次，攻击后移动力归零 |
| 建造改良 | 在己方城市边界内建造改良 | 每城每回合一次（与建筑互斥） |
| 建造建筑 | 在己方城市内建造建筑 | 每城每回合一次（与改良互斥） |
| 招募单位 | 在己方城市招募新单位 | 单位出现在城市格 |
| 城墙射击 | 有城墙的城市执行远程射击 | 每城每回合一次 |

### 3.3 玩家结束回合

- 玩家随时可以点击"结束回合"
- 结束回合后：该玩家不可再执行任何操作，进入观战视角
- 当**所有人类玩家**都结束回合后，自动进入 Phase 3

### 3.4 多人模式的超时机制 [待定]

| 设定 | 说明 |
|------|------|
| 总回合时限 | 每个玩家的总操作时间上限 |
| 超时处理 | 超时后强制结束回合 |
| 断线重连 | 断线玩家自动结束回合，AI 代管可选 |

## 4. Phase 3：AI 行动

AI 在人类全部结束后依次行动。

```
Phase 2 结束（所有人类玩家已结束回合）
        │
        ▼
   AI-1 行动（完成所有操作）
   ├── AI-1 遍历己方所有单位
   ├── 每个单位：评估 → 选择最佳操作 → 执行
   └── AI-1 结束回合
        │
        ▼
   AI-2 行动（同上）
        │
        ▼
   …… 所有 AI 完成
        │
        ▼
   进入 Phase 4
```

> AI 的行动在服务器端快速执行，对玩家表现为一次性结果（玩家不需要等待 AI 一步步走棋，可以一次性看到 AI 的全部操作结果）。

---

## 5. Phase 4：回合结束

```
1. 检查胜利条件
   ├── 某阵营是否消灭了所有敌对阵营？ → 征服胜利
   └── 某玩家是否投降？ → 对方胜利

2. 若无胜利
   ├── 回合数 +1
   └── 进入下一回合（回到 Phase 1）
```

## 6. 城市建造限制

### 6.1 改良建造范围

改良只能建在距市中心 **2 格以内**（含 2 格）的单元格上。

```
距离 0：市中心格（城市所在格，不能建改良）
距离 1：相邻的 6 格
距离 2：再外一圈的 12 格
                  ⬡
               ⬡  ⬡  ⬡
            ⬡  ⬡  🏠  ⬡  ⬡
               ⬡  ⬡  ⬡
                  ⬡
```

- 最大可建改良的格子数：19 格（R=2 范围减去市中心 1 格）
- 城市边界扩张**不扩大改良建造范围**（固定 2 格）
- 超出 2 格的边界单元格仅用于控制领土，不可建改良

> 固定 2 格范围意味着：一座城市最多拥有 19 个改良位。这个上限让玩家需要在"铺更多城"和"把现有城市建设得更强"之间做选择。

### 6.2 建造互斥规则

每座城市每个回合**只能选择以下操作之一**：

| 操作 | 是否消耗本回合建造机会 |
|------|---------------------|
| 建造一个建筑 | ✅ 消耗 |
| 建造一个改良 | ✅ 消耗 |
| 招募一个单位 | ❌ 不消耗（可以同时建造建筑/改良 + 招募） |

即：一回合内，一座城市可以**招募单位 + 建造一个建筑**，或**招募单位 + 建造一个改良**，但不能**同时建造建筑和改良**。

## 7. 完整回合示例（单人模式）

```
回合 5 开始
  │
  Phase 1：自动结算
  ├── 临冬城产出：金币 +14，人力 +10，木材 +4 ...
  ├── 维护费：-6 金币（3 个单位）
  ├── 临冬城 HP 恢复 +5
  ├── 城墙（上次受损回合 3）→ 已 2 回合未受损，继续等待
  ├── 步兵 A：move_points = 3, has_attacked = false
  ├── 弓手 B：move_points = 3, has_attacked = false
  └── buff 计时：无
  │
  Phase 2：玩家行动
  ├── 选中步兵 A → 移动至 (12,8) → 消耗 2 移动力
  ├── 选中弓手 B → 攻击敌方步兵 → 造成 15 伤害
  ├── 选中步兵 A → 攻击已受伤步兵 → 击杀
  ├── 打开临冬城面板 → 建造市场 → 消耗金币 50
  ├── 点击"结束回合"
  │
  Phase 3：AI 行动
  └── AI（西境）完成其所有操作（服务器端快速执行）
  │
  Phase 4：回合结束
  ├── 检查胜利条件：西境仍有城市 → 未达成胜利
  └── 回合数 → 6
```

## 8. 设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 人类玩家同时行动 | 是 | 多人对局不互相等待，缩短等待时间 |
| AI 在人类后轮流行棋 | 是 | AI 快速执行，人类不需要等待 AI 一步步操作 |
| 操作即时生效 | 是 | 服务器处理完即写入 GameState |
| 冲突先到先得 | 是 | 避免复杂的一致性协商算法 |
| 改良建造范围固定 2 格 | 是 | 限制单城发展上限，鼓励多建城市；避免边界扩张导致的管理复杂度 |
| 建筑和改良互斥 | 每回合只能选一个 | 玩家需要做取舍，增加决策深度 |
| 招募不与建筑/改良互斥 | 是 | 保证城市每回合至少可以稳定生产军事单位，不影响游戏节奏 |
